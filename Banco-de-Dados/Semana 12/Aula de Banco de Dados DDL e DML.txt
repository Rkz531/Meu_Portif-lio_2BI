4. Conteúdos essenciais
4.1 O que é SQL?
SQL significa Structured Query Language, ou Linguagem de Consulta Estruturada. Ela é usada para criar estruturas em bancos de dados, inserir informações, consultar registros, atualizar dados e remover dados quando necessário.
Em uma loja virtual, por exemplo, o SQL pode ser usado para criar uma tabela de produtos, cadastrar novos produtos, buscar produtos por categoria, alterar preços e remover registros incorretos.
4.2 Diferença entre DDL e DML
Grupo
Significado
Função principal
Exemplos
DDL
Data Definition Language
Define ou altera a estrutura do banco de dados.
CREATE TABLE, ALTER TABLE, DROP TABLE, CREATE INDEX, DROP INDEX
DML
Data Manipulation Language
Manipula os dados armazenados nas tabelas.
INSERT, SELECT, UPDATE, DELETE


Observação: Uma forma simples de explicar: DDL mexe na estrutura, como criar ou alterar uma tabela. DML mexe nos dados dentro da estrutura, como cadastrar, consultar, atualizar ou excluir registros.
4.3 Analogia para iniciantes
Imagine uma escola com armários para guardar documentos. O DDL seria a ação de construir os armários, definir quantas gavetas existem e colocar etiquetas em cada gaveta. O DML seria a ação de guardar, consultar, trocar ou retirar documentos de dentro desses armários.

INSERT INTO clientes ...
Procurar um cliente cadastrado
SELECT ... FROM clientes
Corrigir o e-mail de um cliente
UPDATE clientes SET email = ...
Remover um cadastro incorreto
DELETE FROM clientes WHERE ...

5. Roteiro introdutório da aula
Nesta aula, você vai aprender a diferença entre criar a estrutura de um banco de dados e manipular os dados que ficam dentro dele. Para facilitar, pense em uma loja virtual: antes de cadastrar produtos, é preciso criar uma tabela para guardar nome, preço, categoria e estoque. Depois disso, é possível inserir, consultar, alterar e excluir produtos.
A parte que cria ou altera a estrutura recebe o nome de DDL. A parte que trabalha com os dados recebe o nome de DML.

7. Exemplos guiados em SQL/MySQL
Os exemplos a seguir podem ser digitados e testados em um ambiente MySQL. Eles usam uma loja fictícia chamada LojaTech.
7.1 Criando e selecionando o banco de dados
DROP DATABASE IF EXISTS loja_tech;
CREATE DATABASE loja_tech;
USE loja_tech;
O comando CREATE DATABASE cria o banco de dados. O comando USE informa qual banco será usado nos próximos comandos.
7.2 Criando uma tabela com CREATE TABLE
CREATE TABLE produtos (
    id_produto INT AUTO_INCREMENT PRIMARY KEY,
    nome_produto VARCHAR(100) NOT NULL,
    categoria VARCHAR(50),
    preco DECIMAL(10,2) NOT NULL,
    quantidade_estoque INT DEFAULT 0 CHECK (quantidade_estoque >= 0)
);
Esse comando cria a tabela produtos. A coluna id_produto identifica cada produto. A coluna nome_produto não pode ficar vazia. O preço usa DECIMAL, indicado para valores monetários.
7.3 Inserindo dados com INSERT
INSERT INTO produtos (nome_produto, categoria, preco, quantidade_estoque)
VALUES
('Mouse Gamer', 'Periféricos', 89.90, 20),
('Teclado Mecânico', 'Periféricos', 249.90, 10),
('Monitor 24 polegadas', 'Monitores', 899.90, 5);
O INSERT adiciona novos registros na tabela. Como id_produto é AUTO_INCREMENT, o banco gera o código automaticamente.
7.4 Consultando dados com SELECT
SELECT * FROM produtos;


SELECT nome_produto, preco
FROM produtos;


SELECT * FROM produtos
WHERE categoria = 'Periféricos';
O SELECT consulta dados. O asterisco (*) mostra todas as colunas. O WHERE filtra registros por uma condição.
7.5 Atualizando registros com UPDATE
UPDATE produtos
SET preco = 219.90
WHERE id_produto = 2;


SELECT * FROM produtos;
Observação: Nunca use UPDATE sem WHERE em atividades reais, porque isso pode alterar todos os registros da tabela.
7.6 Excluindo registros com DELETE
DELETE FROM produtos
WHERE id_produto = 3;


SELECT * FROM produtos;
Observação: Assim como no UPDATE, o DELETE sem WHERE pode apagar todos os registros da tabela.
7.7 Alterando a tabela com ALTER TABLE
ALTER TABLE produtos
ADD COLUMN marca VARCHAR(50);


ALTER TABLE produtos
MODIFY COLUMN nome_produto VARCHAR(150);


ALTER TABLE produtos
DROP COLUMN marca;
O ALTER TABLE modifica a estrutura da tabela. Ele pode adicionar colunas, alterar tipos/tamanhos e remover colunas.
7.8 Criando duas tabelas relacionadas
CREATE TABLE clientes (
    id_cliente INT AUTO_INCREMENT PRIMARY KEY,
    nome_cliente VARCHAR(100) NOT NULL,
    email VARCHAR(100) UNIQUE
);


CREATE TABLE pedidos (
    id_pedido INT AUTO_INCREMENT PRIMARY KEY,
    id_cliente INT,
    data_pedido DATETIME,
    valor_total DECIMAL(10,2),
    FOREIGN KEY (id_cliente) REFERENCES clientes(id_cliente)
);
A tabela pedidos possui id_cliente como chave estrangeira. Isso cria uma relação entre o pedido e o cliente que fez a compra.
7.9 Criando e removendo índices
CREATE INDEX idx_nome_produto
ON produtos (nome_produto);


DROP INDEX idx_nome_produto
ON produtos;
Índices podem melhorar a velocidade de consultas, mas em excesso podem prejudicar inserções, atualizações e exclusões, pois o banco precisa manter os índices atualizados.
8. Dez exercícios guiados com respostas para digitar e testar
Os exercícios desta seção possuem respostas porque fazem parte da prática guiada da aula teórica. A proposta é que o aluno digite, execute, observe o resultado e entenda o funcionamento dos comandos.
Exercício 1 — Criar uma tabela simples
O que o aluno deve praticar: Praticar CREATE TABLE e tipos de dados básicos.
CREATE TABLE categorias (
    id_categoria INT AUTO_INCREMENT PRIMARY KEY,
    nome_categoria VARCHAR(80) NOT NULL
);
Explicação: A tabela categorias foi criada com uma chave primária automática e uma coluna obrigatória para o nome.
Exercício 2 — Inserir registros
O que o aluno deve praticar: Praticar INSERT INTO.
INSERT INTO categorias (nome_categoria)
VALUES
('Informática'),
('Periféricos'),
('Monitores');
Explicação: O comando adiciona três categorias. O id é gerado automaticamente pelo AUTO_INCREMENT.
Exercício 3 — Consultar todos os dados
O que o aluno deve praticar: Praticar SELECT com asterisco.
SELECT * FROM categorias;
Explicação: O SELECT * mostra todas as colunas e todos os registros da tabela categorias.
Exercício 4 — Consultar dados específicos com WHERE
O que o aluno deve praticar: Praticar filtro com WHERE.
SELECT * FROM categorias
WHERE nome_categoria = 'Periféricos';
Explicação: A consulta mostra apenas os registros que atendem à condição definida no WHERE.
Exercício 5 — Atualizar um registro
O que o aluno deve praticar: Praticar UPDATE com WHERE.
UPDATE categorias
SET nome_categoria = 'Acessórios'
WHERE id_categoria = 2;


SELECT * FROM categorias;
Explicação: O registro de id 2 é alterado. O SELECT final ajuda a conferir o resultado.
Exercício 6 — Remover um registro
O que o aluno deve praticar: Praticar DELETE com WHERE.
DELETE FROM categorias
WHERE id_categoria = 3;


SELECT * FROM categorias;
Explicação: O registro de id 3 é removido. O WHERE evita que todos os registros sejam apagados.
Exercício 7 — Criar tabela com chave primária
O que o aluno deve praticar: Praticar PRIMARY KEY e AUTO_INCREMENT.
CREATE TABLE fornecedores (
    id_fornecedor INT AUTO_INCREMENT PRIMARY KEY,
    nome_fornecedor VARCHAR(100) NOT NULL,
    telefone VARCHAR(15)
);
Explicação: A chave primária identifica cada fornecedor de forma única.
Exercício 8 — Criar duas tabelas com chave estrangeira
O que o aluno deve praticar: Praticar relacionamento entre tabelas.
CREATE TABLE departamentos (
    id_departamento INT AUTO_INCREMENT PRIMARY KEY,
    nome_departamento VARCHAR(100) NOT NULL
);


CREATE TABLE funcionarios (
    id_funcionario INT AUTO_INCREMENT PRIMARY KEY,
    nome_funcionario VARCHAR(100) NOT NULL,
    id_departamento INT,
    FOREIGN KEY (id_departamento) REFERENCES departamentos(id_departamento)

Explicação: A tabela funcionarios usa id_departamento como chave estrangeira para se relacionar com departamentos.
Exercício 9 — Alterar uma tabela com ALTER TABLE
O que o aluno deve praticar: Praticar adição de coluna.

Explicação: A tabela fornecedores recebe uma nova coluna chamada email.
Exercício 10 — Criar e remover um índice
O que o aluno deve praticar: Praticar CREATE INDEX e DROP INDEX.
CREATE INDEX idx_nome_fornecedor
ON fornecedores (nome_fornecedor);


DROP INDEX idx_nome_fornecedor
ON fornecedores;
Explicação: O índice é criado para otimizar buscas pelo nome do fornecedor e depois removido.

Tarefa 8: Escrever uma breve reflexão sobre os cuidados necessários ao modificar tabelas em bancos reais.
Ao modificar tabelas em bancos de dados reais, é necessário tomar muito cuidado, pois uma alteração errada pode causar perda de dados, erros no sistema ou problemas em relatórios e integrações. Antes de usar comandos como ALTER TABLE, DROP COLUMN, UPDATE ou DELETE, é importante fazer backup, testar em um ambiente separado e verificar se a coluna ou tabela não está sendo usada por outras partes do sistema. Também é importante planejar bem os tipos de dados para evitar problemas futuros. 

9.6 Perguntas de reflexão
Por que é importante usar o comando ALTER TABLE com cuidado em bancos de dados de produção?
É importante usar ALTER TABLE com cuidado porque ele altera a estrutura da tabela. Em um banco de produção, isso pode afetar sistemas que dependem daquela tabela, causar erros em consultas, remover dados importantes ou deixar o sistema indisponível por um tempo. Por isso, antes de executar esse comando, é recomendado fazer backup, testar antes e verificar os impactos da alteração. 
Como você garantiria a integridade dos dados ao modificar o tipo de uma coluna?
Eu garantiria a integridade dos dados fazendo um backup antes da alteração, verificando se os dados atuais são compatíveis com o novo tipo e testando a mudança em um ambiente de desenvolvimento. Também seria importante conferir se não existem valores inválidos na coluna e, depois da alteração, executar consultas para confirmar se os dados continuaram corretos. 
Quais desafios podem surgir ao remover colunas que podem estar sendo usadas em outros sistemas ou relatórios?
Ao remover uma coluna, podem surgir erros em sistemas, relatórios, consultas SQL, telas e integrações que dependem daquela informação. Além disso, pode ocorrer perda definitiva dos dados armazenados naquela coluna. Por isso, antes de remover uma coluna, é necessário verificar onde ela é usada, avisar a equipe, fazer backup e confirmar se a remoção realmente é segura.

10. Principais erros dos alunos iniciantes
Erro comum
Como evitar
Esquecer ponto e vírgula no final do comando
Conferir se cada comando SQL termina com ;
Usar UPDATE sem WHERE
Sempre verificar qual registro será alterado antes de executar.
Usar DELETE sem WHERE
Executar SELECT antes para conferir o registro que será excluído.
Confundir DDL com DML
Perguntar: estou mexendo na estrutura ou nos dados?
Criar coluna com tipo inadequado
Escolher tipos conforme o dado: texto, número, data ou valor monetário.

Verificar se a coluna é usada por relatórios, sistemas ou integrações.
Criar índices demais
Criar índices apenas em colunas usadas em consultas frequentes.


11. Dicas para evitar problemas em SQL
Antes de alterar ou excluir dados, execute um SELECT para conferir os registros afetados.
Em bancos reais, faça backup antes de alterações importantes.
Teste comandos em ambiente de desenvolvimento antes de usar em produção.
Use nomes claros para tabelas, colunas e índices.
Evite apagar tabelas ou colunas sem verificar dependências.
Lembre-se: DDL altera a estrutura; DML manipula os dados.
12. Resumo do que foi aprendido
